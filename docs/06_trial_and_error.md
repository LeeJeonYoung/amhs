# 06. 시행착오 기록 (Trial & Error)

> "되는 코드"가 아니라 "왜 안 됐고 어떻게 고쳤는지"를 남긴다.
> 면접 방어용 — 각 항목은 **증상 → 원인 → 해결 → 배운 점** 순서. 실제로 겪고 고친 것만 적는다.

---

## A. 무선 통신 (nRF24L01)

### A-1. WiFi/ESP32 대신 nRF24 를 고른 이유
- **고민**: 처음엔 ESP32 + WiFi 로 로봇을 제어하려 했다.
- **문제**: (1) 키트 보드가 UNO 라 ESP32 로 바꾸면 모터쉴드·배선을 다 다시, (2) WiFi 는 AP/IP/재접속 처리가 무겁고 지연이 들쭉날쭉, (3) 납땜 장비가 없음.
- **결정**: **nRF24L01** 채택. 2.4GHz 전용 링크라 지연이 일정하고, 한 모듈이 **6개 파이프**를 네이티브로 지원해 허브 1개가 로봇 3대를 폴링하기에 딱 맞다. 키트 UNO 는 그대로 두고 모듈만 추가.
- **배운 점**: "최신 기술(WiFi)"보다 **문제에 맞는 통신 토폴로지**(중앙 허브 ↔ 다수 노드, 저지연, 결정적 폴링)가 중요. 실제 팹 AMHS 도 컨트롤러↔다수 반송차의 결정적 통신이 핵심이다.

### A-2. SPI 핀 충돌 → 소프트SPI 로 회피
- **증상**: nRF24 하드웨어 SPI(11/12/13핀)와 모터쉴드 핀이 겹쳐 둘 다 오작동.
- **원인**: AFMotor(모터쉴드)와 nRF24 가 같은 디지털 핀을 두고 다툼.
- **해결**: nRF24 를 **소프트SPI**로 돌려 아날로그 포트로 이설 — `CSN=A1, SCK=A2, MOSI=A3, MISO=A4`. `RF24_config.h` 에 `SOFTSPI` 활성화. CE 는 로봇에서 5V 직결(더미 D6)로 핀 절약.
- **배운 점**: 핀 맵 충돌은 배선 문제처럼 보이지만 **버스 중재** 문제다. 소프트SPI 로 버스를 분리해 해결.

### A-3. 간헐적 NO-ACK
- **증상**: 허브가 로봇을 폴링하는데 가끔 `no-ack` 가 뜨고 명령이 씹힘.
- **원인 1**: AA 건전지 **접촉 불량**(살짝 빠져 있었음).
- **원인 2**: 모터 기동 시 전류 급증 → **5V 강하** → 모듈 순간 리셋.
- **해결**: 건전지 홀더 고정, `RF24_PA_LOW`(송신출력 낮춰 전류 부담↓), 허브 펌웨어에 **NO-ACK 20연속 시 경고 + `flush_rx()`**, `robot_id` 검증 추가.
- **배운 점**: 무선 불안정의 1순위 용의자는 전파가 아니라 **전원**. 통신 로그(연속 실패 카운트)를 남겨야 원인이 보인다.

## B. 라인 추종 / 하드웨어 캘리브레이션

### B-1. 센서 극성 (검정이 LOW? HIGH?)
- **증상**: 라인을 따라가야 하는데 라인을 **피해** 달림.
- **원인**: IR 센서가 검정 라인 위에서 LOW 를 내는지 HIGH 를 내는지 키트마다 다르다.
- **해결**: 실측 후 `#define LINE_ON LOW` 로 확정(검정=LOW, 흰=HIGH). 코드 한 줄로 극성을 뒤집을 수 있게 상수화.
- **배운 점**: 센서는 **데이터시트보다 실측**. 극성을 코드 상수로 빼두면 키트가 바뀌어도 1줄 수정.

### B-2. 왼쪽 모터가 거꾸로 돈다
- **증상**: 전진 명령에 한 바퀴는 앞으로, 다른 바퀴는 뒤로 → 제자리 회전.
- **원인**: 왼쪽 모터 배선(IN1/IN2)이 반대로 꽂힘.
- **해결**: 배선을 다시 만지는 대신 **소프트웨어로 방향 반전** — `#define L_FWD BACKWARD`, `L_BWD FORWARD`. 모든 모터 헬퍼가 이 매크로만 쓰게 해 한 곳에서 제어.
- **배운 점**: 물리 배선 실수도 추상화 계층(매크로) 하나로 흡수 가능. 재현·롤백이 쉬운 쪽을 택한다.

### B-3. 오른쪽 모터가 약해서 한쪽으로 쏠림
- **증상**: 직진 명령에도 살짝 한쪽으로 휜다.
- **원인**: 두 모터의 실제 출력이 미세하게 다름(개체차).
- **해결**: `#define TRIM_R 30` — 오른쪽 모터에만 +30 보정. 직진 시 좌우 속도를 비대칭으로.
- **배운 점**: 똑같은 부품도 **개체차**가 있다. 트림값 같은 보정 파라미터를 코드에 두면 현장 튜닝이 빨라진다.

### B-4. 흰 바닥에서 갑자기 멈춤
- **증상**: 라인을 벗어나 흰 바닥에 오면 정지.
- **원인**: 두 센서 모두 라인 없음 → 상태 머신이 애매하게 빠짐 + 감도 포텐셔미터 부적절.
- **해결**: 포텐셔미터로 감도 재조정 + **시작은 직진 모드로** 진입(아래 C-2). 두 센서 OFF 는 "직진"으로 처리.
- **배운 점**: 바닥 대비(흰 바닥 + 19mm 검정 테이프)가 인식률을 좌우. 트랙 재료도 설계 변수다.

### 확정 세팅 (실주행 1대 성공, 2026-06-21)
| 파라미터 | 값 | 의미 |
|---|---|---|
| `LINE_ON` | LOW | 검정=LOW |
| `BASE_SPEED` | 130 | 기본 주행 |
| `SLOW_SPEED` | 20 | 보정 시 안쪽 바퀴 |
| `TURN_SPEED` | 160 | 제자리 선회 |
| `TRIM_R` | +30 | 오른쪽 모터 보정 |
| `L_FWD` | BACKWARD | 왼쪽 모터 방향 반전 |

## C. 교차로 인식 / 회전

### C-1. 바닥 얼룩을 교차로로 오인식
- **증상**: 멀쩡한 직선에서 "교차로 도착"이 떠 멈춤. 로봇을 내려놓는 순간에도 오인식.
- **원인**: 두 센서가 **순간적으로** 동시에 검정을 보면(얼룩, 두꺼운 라인) 교차로로 판정.
- **해결**: **디바운스** — 두 센서 동시 ON 이 `INTERSECT_MS = 50ms` 이상 **지속**돼야 교차로로 인정. 통과 직후 `RENODE_MS = 600ms` 는 재검출 무시.
- **배운 점**: 센서 이벤트는 **시간 필터**로 노이즈를 걸러야 한다. 임계 시간은 주행 속도와 라인 폭의 함수.

### C-2. 시작하자마자 교차로로 빠지는 문제
- **증상**: 로봇을 교차로(노드) 위에 올려두고 출발시키면 곧장 "도착 후 대기" 상태에 갇힘.
- **원인**: 출발 노드 자체가 가로선이라 `RUN` 직후 바로 교차로로 인식.
- **해결**: 출발 명령을 **직진 통과(STRAIGHT)**로 줘서 노드를 빠져나가게(`NUDGE` 단계) 한 뒤 라인추종 진입. 격자 버전에선 서버가 "출발 노드에서의 첫 회전 명령"을 이 용도로 보낸다.
- **배운 점**: "정지 노드 위에서 출발"은 상태 머신의 **초기 조건**을 따로 설계해야 하는 경계 케이스.

### C-3. 90° vs 180° 선회 시간
- **증상**: 좌/우 회전 각도가 안 맞음.
- **해결**: 제자리 선회 시간을 튜닝값으로 — `TURN_MS = 650`(90°), `UTURN_MS = 1300`(180°≈2배). `NUDGE_MS = 150` 으로 먼저 교차로 중심까지 전진 후 선회.
- **배운 점**: 엔코더가 없는 저가 키트에선 **시간 기반 개루프** 회전이 현실적. 대신 바닥/전압이 바뀌면 재튜닝 필요(한계도 명시).

## D. 알고리즘 / 설계 결정 (시뮬 ↔ 실물)

### D-1. ★ 혼잡인지 라우팅이 정적 최단보다 더 나빴다
- **가설**: 간선 부하를 가중치에 반영(혼잡 회피)하면 throughput 이 오를 것이다.
- **실험**: 오프라인 시뮬을 **이동 모델 두 가지**로 돌려 비교.
  - 흐름(유량) 모델: 혼잡 라우팅이 유리.
  - **물리(노드 점유) 모델**: 혼잡 라우팅이 **오히려 나빠짐**(우회가 또 다른 혼잡을 만드는 thrashing).
- **결정**: 실물은 노드 점유 모델이므로 **기본을 정적 최단(static)** 으로. 혼잡 라우팅은 옵션으로만 둠.
- **배운 점**: 알고리즘 우열은 **모델 가정에 종속**된다. nix 시뮬의 "물리모델 비교" 결론과 일치 — 좋아 보이는 최적화가 실제 제약(상호배제) 아래선 손해일 수 있다.

### D-2. 교차로 데드락을 어떻게 풀 것인가
- **문제**: 두 로봇이 서로의 다음 노드를 점유해 둘 다 못 감(상호 대기).
- **해결 3단**: (1) 다음 노드 **예약**(한 노드 한 대), (2) 막히면 점유 노드 **회피 재경로**, (3) 그래도 막히면 인접 빈 노드로 **유턴 후퇴**해 내 노드를 비워 상대를 통과시킴.
- **검증**: 오프라인 시뮬에서 고부하에도 영구 교착 없이 작업이 계속 처리됨(테스트 `test_sim_no_permanent_gridlock`). 같은 로직을 실물 관제 서버(`fleet_grid_server.py`)에 그대로 이식, 3대 교차 시나리오에서 충돌 0·전원 도착 확인.
- **배운 점**: AMHS 의 핵심 난제는 경로 자체가 아니라 **공유 자원(교차로) 경합**. 예약→회피→양보의 계층적 회복이 실용적.

## E. 소프트웨어 구조

### E-1. ★ 세 곳에서 회전 코드가 제각각이었다 (이번에 통일)
- **증상**: 같은 "좌회전"인데 계층마다 번호가 달랐다.
  - 펌웨어(8자): `STRAIGHT=2, LEFT=3, RIGHT=4` (유턴 없음)
  - 기하 모듈: `STRAIGHT=0, RIGHT=1, UTURN=2, LEFT=3`
  - 8자 서버: `GO_THROUGH=2` (회전 개념 자체가 없음)
- **위험**: 서버가 "좌회전"을 보내려는데 펌웨어는 다른 동작을 함 → 디버깅 지옥.
- **해결**: **`navigator.py` 단일 정본** 도입. 기하 회전코드 ↔ 펌웨어 mode 매핑(`FW_MODE`)을 **한 곳에서만** 정의하고, 격자 펌웨어(`robot_grid.ino`)·관제 서버·문서가 모두 이를 참조. 8자 버전(`robot.ino`/`fleet_server.py`)은 검증된 상태라 **건드리지 않고 보존**, 격자용을 새 파일로 분리.
- **배운 점**: 계층 간 **프로토콜의 단일 출처(single source of truth)**가 없으면 "각자 맞다고 믿는" 상태로 어긋난다. 매핑은 코드 한 곳 + 테스트로 고정.

### E-2. 제어 로직과 하드웨어 I/O 분리
- **결정**: 관제 서버의 제어 로직(`GridFleet`)을 시리얼과 분리하고, 전송층을 **실물(serial) / 시뮬(가짜 로봇)** 두 가지로.
- **효과**: 부품이 없거나 트랙을 못 깔아도 **노트북만으로 전체 스택**(경로·예약·회복·LLM)을 결정론적으로 시연·테스트. 단위 테스트가 하드웨어 없이 18개 통과.
- **배운 점**: I/O 경계를 콜백 하나로 좁히면 **테스트 가능성**이 급상승. 실물 디버깅 시간을 코드 단계에서 미리 회수.

## 한 줄 요약 (면접용)
> "통신은 전원 문제였고, 센서는 시간 필터가 답이었고, 알고리즘은 모델 가정에 따라 우열이 뒤집혔고, 버그의 절반은 계층 간 프로토콜 불일치였다. 그래서 **단일 정본 + I/O 분리 + 하드웨어 없는 결정론 테스트**로 구조를 잡았다."
